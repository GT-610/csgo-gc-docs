# 贡献

csgo_gc 仍未完成，因此最有价值的贡献通常是改进正确性、诊断能力、可复现性或文档。

## 使用生成式 AI 贡献

我们不拒绝使用生成式 AI，但是**只有您自己**对您提交的代码负责。我们**不接受任何以 AI 名义签名（sign off）的提交**。

以下行为是不允许的，我们不会接受此类 PR，即使您的 PR 确实包含了问题修复：

```
signed off by Claude
```

即使您使用生成式 AI 开发，也请**以您的名义**提交。我们不会在意您的代码使用何种方式编写，但如果您使用生成式 AI 编写的代码存在问题和风险，**您应对此负责**。

我们建议您在做任何开发之前，先让生成式 AI 阅读仓库内的 AGENTS.md 以了解项目情况。目前大多数 Agent 会自动阅读 AGENTS.md，但我们仍然建议您显式向 AI 要求。


## 适合贡献的方向

- 库存操作边界情况验证。
- 针对损坏库存数据的更安全、更清晰诊断。
- 围绕 RCON 的实时库存编辑工具。
- 物品使用流程的测试或聚焦复现用例。
- Item schema 研究，尤其是较新的完整 CS:GO Item Schema。
- 平台特定安装和构建行为的文档。

## 修改行为前

先判断行为属于哪条路径：

- 玩家库存和物品行为通常在 `inventory.cpp` 和 `gc_client.cpp`。
- 商店行为通常从 `gc_client.cpp` 开始，并使用 `price_sheet.txt`。
- 服务器互操作通常涉及 `gc_server.cpp` 和 networking 文件。
- Steam API 拦截通常属于 `steam_hook.cpp`。
- 配置变更应同时更新 `config.cpp`、`config.h`、示例配置和用户文档。

## 兼容性预期

优先选择小而聚焦的变更。许多流程是在模拟特定的 CS:GO GC 消息形状，因此看起来很局部的改动也可能影响库存缓存更新、UI 通知或专用服务器行为。

添加新的用户可见选项或 RCON 参数时，请在同一个变更中更新文档。

## 文档风格

文档需要照顾两类读者：

- 玩家需要安全的安装、配置和排障步骤。
- 开发者需要源码入口、消息流、数据格式和已知限制。

不要把实验性行为写成稳定承诺。
